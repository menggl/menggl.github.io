JAVA虚拟机之类加载器

发表于 2018-08-06 | 分类于 经典

java虚拟机包括：
类加载器：加载类用的
执行引擎：执行java代码用的
运行时数据区：相当于操作系统内存，保存虚拟机运行时的缓存数据用的，是一块需要优化的部分

类加载器将java的.class文件按照规范加载到运行时数据区的方法区中，以供执行引擎解释执行

每个类加载器将class文件加载完之后，都会将(包名+类名)当做一个命名空间保存起来，也就是说每一个类加载器只能加载一份（包名+类名）的文件
不同的类加载器可以加载相同的(包名+类名)文件

java类加载器是分层级的，最顶级的类加载器是BootstrapClassLoader，其次的类加载器是ExtClassLoader，最后才是AppClassLoader
结构如下所示：
–BootstrapClassLoader 加载$JAVA_HOME$/lib/rt.jar下的类
—-ExtClassLoader 加载$JAVA_HOME$/lib/ext.jar下的类
——AppClassLoader
——–URLClassLoader spring中的类加载器
———-WebappClassLoader 在tomcat中部署的每个项目都会创建一个类加载器，以保证加载的相同命名空间（不同项目、不同实现）的加载类不同，以实现隔离项目冲突的目的
各个类加载器之间是组合的关系，不是继承的关系

双亲委派模型
当一个classLoader想加载一个class文件的时候，首先让自己的父加载器加载这个class（要站在巨人的肩膀上），如果父类加载器加载失败，则只能自己去加载这个class了
下面是ClassLoader.loadClass的代码实现

测试类加载器加载顺序
创建下面的类，并将编译好的class文件及命名空间文件夹放到D:\temp目录下面
public class HelloLoader {
public void print() {
System.out.println(“I am in BootstrapClassLoader!”);
}
}
然后改变HelloLoader为：
public class HelloLoader {
public void print() {
System.out.println(“I am in AppClassLoader!”);
}
}
开始测试：定义下面这个类文件
public class FindClassOrder {
public static void main(String[] args) {
HelloLoader helloLoader = new HelloLoader();
helloLoader.print();
}
}
正常情况下回输出：I am in AppClassLoader!
添加虚拟机运行参数：-Xbootclasspath/a:D:\temp（添加BootstrapClassLoader加载类路径的范围D:\temp）
再运行时输出变成了：I am in BootstrapClassLoader!

即使配置了运行参数：-Xbootclasspath/a:D:\temp
如果AppClassLoader强行加载HelloLoader，也不会使用到父加载器加载的类定义
import java.io.File;
import java.io.FileInputStream;
import java.lang.reflect.Method;
import org.apache.commons.io.IOUtils;
public class FindClassOrder {
public static void main(String[] args) throws Exception {
// 使用appClassLoader强行加载HelloLoader这个class
ClassLoader classLoader = FindClassOrder.class.getClassLoader();
byte[] bHelloLoader = loadClassBytes(“D:\tmp\com\test\classloader\HelloLoader.class”);
Method defineClassMethod = ClassLoader.class.getDeclaredMethod(“defineClass”, byte[].class,int.class,int.class);
defineClassMethod.setAccessible(true);
defineClassMethod.invoke(classLoader, new Object[]{bHelloLoader,0,bHelloLoader.length});
defineClassMethod.setAccessible(false);

    // 当前AppClassLoader强行加载了HelloLoader这个类，就不会再使用父加载器加载的这个类了
    HelloLoader helloLoader = new HelloLoader();
    helloLoader.print();

}
public static byte[] loadClassBytes(String filePath) throws Exception {
    return IOUtils.toByteArray(new FileInputStream(new File(filePath)));
}

}

双亲委托模型的弊端：
父类加载器无法访问子类加载器的对象。比如父类加载器定义一个接口，同时绑定一个工厂方法，由于接口实现类不在父加载器中，所以无法访问和创建实例
